# Опис схеми бази даних

Система використовує **PostgreSQL**. Схема визначена у `server/src/db/schema.sql`.

## Користувацькі типи (ENUM)

| Тип | Значення |
|-----|----------|
| `user_role` | `admin`, `head`, `member` |
| `project_status` | `planned`, `active`, `on_hold`, `completed`, `cancelled` |
| `task_status` | `todo`, `in_progress`, `review`, `done` |
| `task_priority` | `low`, `medium`, `high`, `critical` |
| `document_status` | `draft`, `on_review`, `approved`, `rejected` |
| `approval_decision` | `pending`, `approved`, `rejected` |

## Таблиці

### organizations — студентські організації
Базова сутність. Об'єднує користувачів та проєкти.

### users — користувачі
Зберігає облікові записи з хешем пароля (bcrypt) та роллю. Належить до однієї організації.

### projects — проєкти
Проєкт належить організації, має власника (`owner_id`), статус, дати початку/завершення.

### memberships — членство в проєктах
Зв'язок **багато-до-багатьох** між користувачами та проєктами з роллю в проєкті (`role_label`). Унікальна пара (user_id, project_id).

### tasks — задачі
Задачі прив'язані до проєкту, мають виконавця, статус (для Kanban-дошки), пріоритет, дедлайн.

### documents — документи
Центральна сутність документообігу. Має тип, зміст, статус та автора. Може бути прив'язана до проєкту.

### document_approvals — маршрут узгодження
Послідовність кроків погодження документа (`step_order`). Кожен крок — погоджувач і його рішення (`decision`). Документ вважається затвердженим, коли всі кроки `approved`; відхиленим — щойно будь-який крок `rejected`.

### document_history — історія документа
Журнал дій над документом (створення, відправлення, рішення) для аудиту.

### comments — коментарі
Коментарі до документів або задач.

### document_templates — шаблони документів
Готові заготовки (наказ, кошторис, звіт, заявка) з типом і вмістом для швидкого створення документів.

### attachments — прикріплені файли
Файли, завантажені до документа (зберігаються на диску в `server/uploads`, у БД — метадані: ім'я, MIME-тип, розмір, хто завантажив).

### notifications — сповіщення
In-app сповіщення користувачів (запит на погодження, затвердження, відхилення) з прапором прочитання та посиланням. Дублюються email-розсилкою.

### activity_log — журнал активності
Стрічка подій по сутностях (`document`, `project`, `task`): хто, коли і що зробив.

### Делегування (поле `delegated_from` у document_approvals)
Погоджувач може передати свій крок іншій особі: `approver_id` замінюється на нового погоджувача, а `delegated_from` фіксує, хто делегував.

## Логіка документообігу

```
draft ──submit──▶ on_review ──(усі approved)──▶ approved
   ▲                  │
   │                  └──(будь-який rejected)──▶ rejected ──edit──▶ (можна повторно submit)
   └──────────────── edit ────────────────────────────┘
```

1. Автор створює документ у статусі `draft`.
2. Автор відправляє на узгодження, задаючи список погоджувачів (маршрут).
3. Кожен погоджувач приймає рішення (погодити / відхилити).
4. Коли всі погодили — документ `approved`. Якщо хтось відхилив — `rejected` (автор може виправити й відправити знову).

Усі дії фіксуються в `document_history`.
